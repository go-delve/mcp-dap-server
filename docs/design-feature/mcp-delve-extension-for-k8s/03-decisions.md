# Design Decisions: MCP Delve Extension для Kubernetes

## Decisions

| # | Decision | Choice | Alternatives Considered | Rationale |
|---|----------|--------|-------------------------|-----------|
| 1 | Имя binary в форке | Оставить `mcp-dap-server` (как в upstream), функциональность активировать через CLI-флаг `--connect` | Переименовать в `mcp-dap-server-k8s` или `mcp-dap-kov` (как в исходной заявке) | Сохранение имени облегчает upstream PR для ступени 1 (ConnectBackend): изменения в `main.go` минимальны — только регистрация нового флага. Upstream-пользователи не увидят поведенческих изменений при обновлении, пока не используют `--connect`. Совместимость — ключевой аргумент. |
| 2 | Ответственность за `kubectl port-forward` | Вне Go-бинаря: bash-wrapper `dlv-k8s-mcp.sh` держит port-forward в retry-loop | Встроить Kubernetes Go-клиент в MCP-сервер и управлять port-forward из Go | Separation of concerns: shell владеет сетью, Go владеет DAP. Снижает surface для upstream PR (в upstream port-forward логика никому не нужна). Минус — Linux-only (bash, nc), но Windows/macOS — out of scope по §9.4 design-doc. |
| 3 | DAP handshake против `dlv --headless` | `InitializeRequest(adapterID="go")` → `AttachRequest{mode: "remote"}` → `SetBreakpoints*` → `ConfigurationDoneRequest` | `AttachRequest{mode: "local", processId: N}` с сохранением PID | `mode: "remote"` — официальный путь для `dlv --headless --accept-multiclient` (Delve DAP README, vscode-go docs). PID undocumented для headless scenario: Delve сам управляет процессом через `exec /binary --continue`. Delve ≥ v1.7.3 требуется — у актуальных Delve v1.25.x выполняется. |
| 4 | Concurrency в `DAPClient` при reconnect | Добавить `sync.Mutex` в `DAPClient` + `atomic.Bool stale` + `chan struct{}` signal; фоновая goroutine для Redial; `reinitialize` держит `ds.mu` на весь lifetime (см. ADR-13) | (A) Полный переход к event-driven reader (одна goroutine читает, диспатчит в response-map + event-channels). (B) Per-request response-channel map при одном reader-goroutine (middle ground). (C) `sync.RWMutex` только вокруг swap `rwc`, без mu-защищённого reinitialize. | (A) ломает совместимость с upstream: рефактор `readAndValidateResponse`/`readTypedResponse` (tools.go:247-312, ~100 строк), trivial merge-conflicts на каждом upstream-bump. (B) лучше (A), но всё ещё трогает существующие хелперы. (C) — недостаточно: без mu на reinitialize получаем race на `ds.breakpoints` между reconnectLoop (чтение при re-apply) и tool-методом `breakpoint` (запись). Наш вариант: mutex в `DAPClient` защищает swap `rwc`; `reinitialize` берёт `ds.mu` и держит её весь network-exchange — tool-methods ждут на mu (это нормально, они всё равно получили бы ErrConnectionStale). **Trade-off**: если reinit длится 5-10 сек, следующий tool-call подождёт эти секунды; но Claude в это время ничего полезного сделать не может. |
| 5 | Breakpoints persistence storage | `map[string][]dap.SourceBreakpoint` + `[]string` для function BPs в `debuggerSession`, обновляется на каждом `breakpoint`/`clear-breakpoints` tool call | Persist на диск (для переживания рестарта MCP-сервера) | Disk persistence — overkill: при exit Claude-сессии debuggee тоже исчезает (pod может жить, но user-intent отладки закончен). Memory-only storage переживает только pod-restart, которого достаточно. |
| 6 | Capability-gating для нового tool `reconnect` | Регистрировать безусловно в session-tools (аналогично `continue`, `step`, `pause`) | Gate по `ds.backend == *ConnectBackend` — имеет смысл только для remote-attach сценария | Проще для пользователя (tool всегда виден в MCP tool list). При `force=true` с `SpawnBackend` — `backend.Redial` вернёт ошибку "not supported", MCP вернёт её Claude'у как tool error. Это понятнее, чем скрытый tool. |
| 7 | Advisory lock в bash-wrapper'е | Не добавляем | `flock` на файле (например `/tmp/dlv-k8s-$NS-$SVC.lock`) для enforcement'а "один инженер на один debuggee" | `--accept-multiclient` в Delve прямо разрешает concurrent клиентов; breakpoint overwrite — trade-off multiclient'а, это ожидаемо. Advisory lock усложняет wrapper ради социальной конвенции, которую лучше решать культурой команды. |
| 8 | Observability | Лог на stderr wrapper'а (`[dlv-k8s-mcp HH:MM:SS] ...`) + файловый лог MCP-сервера (`$TMPDIR/mcp-dap-server.log`) | Prometheus metrics (reconnect count, backoff histogram, session uptime) | Dev-tool на рабочем ноутбуке — никто не будет собирать метрики. Logs достаточно для debugging самого MCP. |
| 9 | Binary-флаг vs environment variable | Оба: CLI `--connect <addr>` **и** env `DAP_CONNECT_ADDR`; при обоих — CLI приоритетнее | Только CLI или только env | Bash-wrapper передаёт через CLI (читает env и `exec mcp-dap-server --connect "$addr"`); разработчик, запускающий MCP напрямую (вне wrapper'а) — может удобнее env. Оба — стандарт для 12-factor apps. |
| 10 | Где объявлен `Redial(ctx)` | **Отдельный interface `Redialer`**, объявленный в `dap.go`: `type Redialer interface { Redial(ctx context.Context) (io.ReadWriteCloser, error) }`. `DAPClient.reconnectLoop` делает `if r, ok := backend.(Redialer); ok { ... }`. `ConnectBackend` реализует `Redialer`, `delveBackend` / `gdbBackend` — не реализуют (ничего не добавляется). | (A) Добавить метод в существующий `DebuggerBackend` (interface widening, стаб-методы "not supported" в delve/gdb backends). | (A) нарушает ISP (Interface Segregation): `Redial` — концерн транспорта, не конфигурации debugger'а. Widening `DebuggerBackend` заставит upstream PR обсуждать изменение public interface'а — это блокер. С отдельным `Redialer`: upstream PR становится **additive-only** (добавляются новые файлы `connect_backend.go` + `redialer.go`, существующие `backend.go`/delve/gdb не трогаются). Клиентский код (`reconnectLoop`) использует Go-идиоматичный optional interface через type-assertion — тот же паттерн, что `io.WriterTo`, `io.StringWriter`. |
| 11 | Режим сохранения `seq` при reconnect | Не сбрасывается | Сбрасывать в 1 после `replaceConn` | После нового `Initialize` новые запросы получают incremented `seq`; старые seq-ы уже не в эфире. Матчинг по `request_seq` работает корректно в обе стороны. Сбрасывать — лишнее, и рискует конфликт с in-flight responses. |
| 12 | Единый `ctx` для shutdown `reconnectLoop` | `context.Background()` c cancel-функцией, хранится в `DAPClient`; `Close()` вызывает cancel | `ctx.Context` передаётся из вызывающего tool-метода | Tool-methods имеют короткий lifetime; `reconnectLoop` должен жить весь lifetime `DAPClient`. Его cancel контролируется только `Close()` (в `cleanup()` сессии). Чистый lifecycle. |
| 13 | `reinitialize` lock-ordering | `reinitialize` захватывает `ds.mu` перед серией DAP-запросов и держит всю последовательность (Initialize → Attach → SetBreakpoints × N → ConfigurationDone). Иерархия lock-ordering: **всегда `ds.mu` → `DAPClient.mu`, никогда наоборот.** Tool-methods и reconnectLoop→reinitialize используют одну и ту же последовательность (первый лочит ds.mu, потом DAP-запросы берут DAPClient.mu). | (A) Snapshot-and-release: `reinitialize` читает `ds.breakpoints` под `ds.mu`, отпускает, работает с snapshot, никак не сериализуется с параллельными tool-методами. (B) Отдельный `ds.bpMu` только для map. | (A) — race: если между snapshot'ом и концом reinit пользователь добавит BP через tool, Delve его не увидит после ConfigurationDone до следующего tool-call'а — confusing UX. (B) — overhead нового mutex'а + lock-ordering между двумя mu внутри debuggerSession. Наш вариант простой: строгая иерархия, нет двойных locks, tool-method ждёт reinit (обычно < 1s). |
| 14 | Partial failure в `reinitialize` (например, 2/3 SetBreakpoints упали) | `reinitialize` возвращает error → `reconnectLoop` оставляет `stale=true`, пытается заново на следующем тикре backoff'а. В тот же reconnect attempt retry не делаем — дёшево стартануть с чистого Initialize. | Continue-on-error внутри reinitialize (best-effort: сообщить Claude про BP-потери). | Retry с нуля — проще и корректнее: после Initialize в Delve BP-состояние пусто, мы всегда восстанавливаем из `ds.breakpoints`. Best-effort усложняет семантику reporting. Добавляем лог `log.Printf("reinitialize: N of M breakpoints applied, retrying"`) для observability. |
| 15 | Observability длительного reconnect (ImagePullBackOff) | `DAPClient` хранит `reconnectAttempts atomic.Uint32` и `lastReconnectError atomic.Value`. `reconnect` tool (при любом status-response) добавляет `attempts`, `last_error` поля. Лог wrapper'а печатает каждую попытку. | (A) Новый отдельный MCP tool `debugger-status`. (B) Периодический status event через MCP notifications. | (A) — дополнительная поверхность API, лишне. (B) — MCP notifications не поддерживаются всеми клиентами. Встраивание в `reconnect` response — минимальный change. |
| 16 | MCP stdio backpressure при длительном reconnect | Tool-methods получают ошибку `ErrConnectionStale` сразу (через `DAPClient.send` pre-check), без ожидания `ds.mu`. Queue tool-calls на stdin MCP обрабатывается Claude-клиентом, не нашей стороной. | Блокирующий wait на `ds.mu` (tool-call зависает пока не восстановится). | Блокировка на минутах создаёт ложное впечатление что MCP-сервер завис, и портит Claude-side timeout поведение. Fast-fail с clear error message лучше: Claude видит structured error, может либо retry либо `reconnect` tool вызвать. **Исключение**: `reconnect` tool сам по себе блокируется sync-wait (это по design). |
| 17 | `reconnect(force=true)` при активной reconnect-loop | Semantically no-op (`markStale` — idempotent CAS, второй вызов не посылает дополнительный сигнал в buffered-1 канал). Wait-loop в tool-handler'е всё равно читает `stale` и ждёт healthy. | Явный cancel-and-restart текущего reconnect attempt. | Cancel-and-restart может привести к потере частично завершённого reinitialize → уход в worse-state. Idempotency — защитная стратегия. Для observability tool добавляет поле `already_reconnecting: true` в response. |
| 18 | Default значение `DLV_RECONNECT_INTERVAL` (bash retry sleep) | `2` секунды (NFR-7) | `1` (агрессивнее) или `5` (щадяще к API server'у) | 2s — золотая середина: достаточно для kubelet-recover после single pod-delete, но не DoS'ит API server при затяжных проблемах. При необходимости overridable через env. |

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| Upstream не примет PR ConnectBackend | Medium (придётся вечно поддерживать форк) | Ступень 1 (ConnectBackend) максимально изолирована: новый файл `connect_backend.go` + минимальный патч в `main.go`. Чистый diff. Подаём PR сразу после ступени 2. Если отклонят — жизнь с форком терпимая, т.к. это ~100 строк кода. |
| Delve < v1.7.3 внутри pod'а | Low (при использовании свежих образов) | Health-check в wrapper'е: `kubectl exec <pod> -- dlv version` и парсинг. При устаревшей версии — вывести сообщение и exit с инструкцией обновить devel-образ. |
| `kubectl port-forward` sleep 2s слишком короткий на медленной сети | Low | Параметр `DLV_RECONNECT_INTERVAL` (default 2), можно поднять до 5 через env. Exponential backoff в bash — не делаем, держим просто. |
| Breakpoint drift после code change (rebuild с новым layout) | Medium (UX-риск, не correctness) | Документируем в README форка: "после pod restart с пересборкой проверьте, что breakpoint всё ещё на нужном stmt". |
| Concurrent DAP client overwrite наших breakpoints | Low (single-user сценарий) | Документируем как known limitation в README. На будущее — можно думать про merge-strategy с breakpoint'ами DAP-адаптера через `BreakpointLocationsRequest` (но не в MVP). |
| Goroutine leak в `reconnectLoop` | Medium (если `Close()` не вызван) | `defer ds.cleanup()` в `main.go:43` гарантирует Close. Unit-test на `cleanup` проверяет завершение goroutine (через sync.WaitGroup). |
| Deadlock между `ds.mu` и `DAPClient.mu` | High (если случится — hang навсегда) | Жёсткая иерархия lock ordering: **всегда `ds.mu` → `DAPClient.mu`, никогда наоборот.** `reinitialize` лочит `ds.mu` первой строкой, затем делает DAP-запросы через `DAPClient.send` (который внутри берёт `DAPClient.mu` на короткое время — swap-rwc). Tool-methods используют ту же последовательность. ADR-13 фиксирует это явно. Unit-test с race detector на sequence "concurrent breakpoint tool + reconnect-with-reinit". |

## Open Questions (все разрешены после исследования)

- [x] **Attach mode + PID для Delve headless** → разрешено: `AttachRequest{mode: "remote"}` без PID (§9.1 design-doc).
- [x] **Launch mode в reinitialize** → разрешено: всегда remote-attach для ConnectBackend (§9.1 design-doc).
- [x] **Advisory lock в wrapper'е** → не добавляем (ADR-7 выше, §9.2 design-doc).
- [x] **Prometheus метрики** → нет (ADR-8 выше, §9.3 design-doc).
- [x] **Windows/macOS поддержка** → не в MVP (§9.4 design-doc).
- [x] **Reader architecture (synchronous vs event-driven)** → синхронный с mutex'ом (ADR-4 выше).
- [x] **`seq` reset** → не сбрасывать (ADR-11 выше).
- [x] **`Redial` interface design** → в существующем `DebuggerBackend` (ADR-10 выше).

## New Questions (для обсуждения в ходе реализации)

- [ ] **Shouldn't `reconnect` tool be async-by-default?** Текущее решение — sync wait до 30s. Если Claude-клиент имеет свой timeout меньше — проблема. Альтернатива: tool сразу возвращает `{"status": "reconnecting"}`, Claude сам вызовет его снова для проверки. **Планируется решить в ходе smoke-test на реальном pod restart.**
- [ ] **В каком файле объявить `connect_backend.go`?** Если upstream монопакет `main`, то просто новый файл рядом с `backend.go`. Если будущее — движение к `pkg/backend/` package — решать в момент PR-preparation.
- [ ] **Нужен ли health-check endpoint для wrapper'а?** Сейчас `nc -z` — проверка TCP accept. Можно дополнить `dlv version` через kubectl exec. Считаем, что первое tool-call покажет проблемы — явного health-check не добавляем в MVP.
- [ ] **Structured error codes для `ErrConnectionStale` и related** — `"connection stale"` как строка удобна для human, но Claude-prompt'ы могут хуже matches'ить её. Альтернатива — MCP error object с `code` field. В MCP SDK `go-sdk` проверить support для custom error codes (см. N4 from review).
- [ ] **Wait-loop polling interval в `reconnect` tool**. Текущее решение — 100ms. Если 100ms слишком часто — можно перейти к channel-based notification (`stale` не atomic.Bool, а `chan bool`). Оставляем polling на MVP, меняем если покажет себя плохо в практике.
